## Thread 1: How To Verify Any Algorand ASA

```text
1/ Asset research starts with the ASA ID.

Before reading a project summary, verify the asset object itself.

For Algorand ASAs, that means name, unit, decimals, total supply, URL, creator, and control fields.

2/ The ASA ID is the anchor.

If the asset ID is wrong, every later claim can be wrong.

Record the ID, source, and capture date.

3/ Convert supply correctly.

Display supply = raw supply / 10^decimals.

Decimals matter. A supply claim without decimals is incomplete.

4/ Check control fields.

Manager.
Reserve.
Freeze.
Clawback.

If a project says controls are cleared, verify the current fields and review history where possible.

5/ Separate supply categories.

Total supply is not the same as circulating supply.
Burned supply is not the same as inaccessible-account supply.
Locked supply needs lock evidence.

6/ Good documentation marks unknowns.

`Needs verification` is not a weakness. It is how public review stays honest.

Educational only. Verify public sources before relying on any asset claim.
```

## Thread 2: Tokenomics Is Not One Number

```text
1/ Tokenomics is not one number.

A useful tokenomics review separates:

- total supply
- decimals
- direct burns
- inaccessible accounts
- locks
- liquidity
- circulating methodology

2/ Total supply comes from the asset record.

For an Algorand ASA, check public indexers and explorers.

3/ Direct burns need evidence.

Do not treat a page-displayed burn number as verified unless the burn address, transaction set, and methodology are public.

4/ Inaccessible accounts are different.

If an account is believed unavailable, that may matter for methodology, but it should not be merged with direct burns without review.

5/ Locked supply needs lock evidence.

Labels are useful leads. They are not final proof.

6/ Better wording:

Observed on [date].
Source: [link].
Status: Needs verification.

That is stronger than pretending uncertainty is gone.
```

## Thread 3: Market Snapshots

```text
1/ Market data expires quickly.

Liquidity, TVL, volume, price, and routes should be treated as dated snapshots unless refreshed.

2/ A good market snapshot includes:

- source
- capture date
- observed value
- method
- limitation
- status

3/ Write:

"Liquidity observed on [date] from [source]. Snapshot only."

Do not write:

"Liquidity is [value]" unless it was just refreshed.

4/ Why this matters:

Old market numbers can mislead people even when they were accurate at capture time.

5/ ProfitNet's standard:

Date it.
Source it.
Mark the limitation.
Recheck before relying on it.
```

## Thread 4: Contribution Credits Explained Safely

```text
1/ PNET Community Contribution Protocol framing:

App-local contribution credits for access and reputation workflows.

Not deposits.
Not yield.
Not passive income.
Not guaranteed value.

2/ App-local means the credits are records inside an application.

They are not PNET tokens.

3/ Non-custodial means the application should not take custody of user PNET.

No deposit requirement should be implied.

4/ A public deployment should publish:

- app ID
- source commit
- ABI
- deployment tx
- admin policy
- receipt policy
- audit status

5/ Until those records exist, status should remain:

TestNet starter only.
Not MainNet.
Not audited.
Not live.
```

## Thread 5: What A Good Asset Dossier Does

```text
1/ A good asset dossier is not a pitch deck.

It should help skeptics check the record.

2/ It should include:

- asset facts
- source links
- tokenomics claims
- status labels
- market snapshots
- risks
- changelog
- verification steps

3/ It should not include:

- private keys
- wallet screenshots
- signer logic
- deposit instructions
- trading automation
- guarantee language

4/ The best documentation habit:

Separate claim from fact.

5/ If the evidence is incomplete, say so.

`Needs verification` protects the reader and the project.
```

## Thread 6: Correction Culture

```text
1/ Transparent projects publish corrections.

Silently deleting a bad claim makes the record weaker.

2/ A good correction includes:

- affected claim
- corrected wording
- source
- date
- reason
- status

3/ Correction language:

"We previously described [claim]. After review, the correct status is [status]. The current evidence is [source]."

4/ This is not a failure.

It is how public records get stronger.
```
